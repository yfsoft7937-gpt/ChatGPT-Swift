# ChatGPT Swift v0.3.0

> 一个针对 ChatGPT 长对话页面的浏览器扩展 - 解决长对话Session中ChatGPT网页响应慢、打字卡顿等性能问题：目标是在不影响正常对话能力的前提下，把历史消息从“全量真实 DOM”改造成“窗口化 DOM”，从而降低长会话下的样式重算、布局和滚动开销。

## 1. 插件解决什么问题

在超长对话页面中，浏览器卡顿通常并不主要来自模型本身，而是来自前端页面渲染压力，例如：

- 历史消息 DOM 节点过多
- 输入时触发大范围 `Recalculate Style` 与 `Layout`
- 滚动时离屏历史内容仍参与渲染
- 长代码块、表格、图片等富内容使页面更重

这个扩展的思路不是修改 ChatGPT 的服务端逻辑，也不是改模型上下文，而是只优化 **显示层**：

- 让视口附近消息继续保留为真实 DOM
- 让远离视口的历史消息暂时从 DOM 中移除
- 用等高 placeholder 占位，保持滚动高度大致稳定
- 需要时再把真实消息节点恢复回来

---

## 2. 核心功能

### 2.1 历史消息窗口化
只保留视口附近和少量缓冲区内的消息为真实 DOM。更远的历史消息会被替换为占位块。

### 2.2 点击恢复
被虚拟化的消息可以点击占位块恢复为真实消息，便于查看和复制。

### 2.3 输入保护期
用户刚输入后的一小段时间内，插件会暂停虚拟化，避免输入和页面重排相互抢主线程。

### 2.4 Debug Overlay
页面右上角有调试面板，用来实时显示：

- 是否启用
- 当前识别到的消息数
- live / virtual 数量
- 更新次数
- 当前原因状态
- detector 命中方式
- mode 模式

### 2.5 配置面板
支持通过弹窗或选项页调整参数，例如：

- overscan
- 最小触发消息数
- 最近消息保留数量
- 输入后冻结时间
- 是否只启用安全模式
- 是否显示调试面板

### 2.6 占位块可恢复映射
v3 修复了早期版本中“被 virtualize 的消息失联”的问题。现在 placeholder 和原消息 meta 会保持映射，滚回或点击时可以恢复。

---

## 3. 插件不会做什么

为了降低风险，这个插件 **不会**：

- 修改 ChatGPT 的模型请求
- 修改会话内容
- 修改 OpenAI 服务端上下文
- 破解、绕过或干预官方限制
- 改写 ChatGPT 内部业务 bundle 的私有逻辑
- 长期保存你的消息内容到插件外部存储

它只做 **前端显示层优化**。

---

## 4. 适用范围

适用场景：

- 长对话滚动明显变慢
- 打字卡顿
- 页面下滑掉帧
- 展开长代码块后卡顿明显
- 旧消息很多时，页面交互变得迟钝

不适合或需谨慎的场景：

- 你依赖浏览器 `Ctrl/Cmd + F` 在整页历史消息里全文搜索
- 你使用某些依赖完整 DOM 的翻译/划词/导出类扩展
- 你需要所有历史消息始终可被直接选择和复制

因为被虚拟化的消息在该时刻 **不在 DOM 中**，虽然通常仍在插件内存中。

## 5. 安装方式

### Chrome / Edge 加载方式
1. 打开 `chrome://extensions` 或 `edge://extensions`
2. 打开“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择插件目录：`chatgpt-windowed-dom-extension-v3`
5. 刷新已打开的 ChatGPT 页面

## 7. 实现原理

## 7.1 总体架构

插件主要在 content script 中工作，流程大致是：

1. 进入 ChatGPT 页面后，识别消息节点
2. 为每条消息建立内部 `meta`
3. 根据视口位置决定哪些消息保留 live，哪些消息改为 virtual
4. 对 virtual 消息，用 placeholder 替换真实节点
5. 滚动或点击时，把需要恢复的消息重新放回 DOM
6. 通过调试面板显示当前状态

---

## 7.2 live / virtual 的含义

每条消息在内部有两种状态：

- `live`：真实消息节点在 DOM 中
- `virtual`：真实消息节点从 DOM 中移除，DOM 中只保留 placeholder

注意：

- `virtual` 不等于删除内容
- v3 中通常仍保留原始节点对象在内存状态里
- 点击恢复时，会直接把原消息节点放回 DOM

---

## 7.3 消息识别

为了降低对 ChatGPT 私有 class 的依赖，插件优先使用较稳的识别方式，例如：

- `main article`
- `[data-message-author-role]`
- `[data-testid^="conversation-turn-"]`

如果一种方式找不到消息，会退回到其他启发式方式。

调试面板中的 `detector` 字段，就是当前命中的识别策略，例如：

- `author-role`
- `main-article`

---

## 7.4 窗口化策略

插件不会盲目删除旧消息，而是维护一个“保活窗口”。

保活窗口通常由三部分组成：

1. **当前视口中可见的消息**
2. **上下两侧的 overscan 缓冲消息**
3. **最近若干条消息**（例如最后几条，避免影响正在对话的底部区域）

简化理解：

```text
live = 可见消息 + 上下 overscan + 最近消息保留
virtual = 其余历史消息
```

这样可以兼顾：

- 滚动顺畅
- 输入区稳定
- 新消息区域不被过早虚拟化

---

## 7.5 placeholder 占位机制

当某条消息需要 virtualize 时，插件会：

1. 测量该消息高度
2. 创建一个等高 placeholder
3. 用 placeholder 替换消息节点

这样可以尽量维持滚动高度和位置，不让页面突然塌缩。

placeholder 具有以下特点：

- 有可视化提示，例如 “History message virtualized”
- 具备点击恢复能力
- 持有原消息 `id` 的映射信息

---

## 7.6 恢复机制

当某条消息需要恢复时：

- 自动恢复：滚动到视口附近后恢复
- 手动恢复：点击 placeholder 恢复

恢复不是重新请求服务器，而是把插件保存的原消息节点重新插回去。

---

## 7.7 输入保护期

用户在输入框中打字时，长会话页面本来就容易触发大量样式重算与布局。

为了避免插件自身进一步放大卡顿，插件会在输入后进入短暂冻结期：

- 暂停虚拟化更新
- 暂停批量替换
- 等用户输入停顿一小段时间再恢复更新

调试面板里常见的 `reason: post-input` 就表示处于输入保护期。

---

## 7.8 为什么 v3 比早期版本稳

v3 修复了几个关键问题：

### 修复一：忽略插件自身的 DOM 替换
早期版本把 `replaceWith()` 引起的 mutation 也当成外部变化，结果触发错误 rescan。

v3 会屏蔽插件自身的替换操作，避免 observer 把内部变更误判为页面内容变化。

### 修复二：保留 virtual meta
早期版本在重新扫描时只保留仍在 DOM 里的 live 节点，导致 virtual 消息的状态丢失。

v3 会保留已 virtual 的 meta，并维护 placeholder 与原节点的对应关系。

### 修复三：支持点击恢复
placeholder 不再只是灰块，而是可以点击恢复消息。

### 修复四：移除导致高度膨胀的做法
早期版本把 `content-visibility` 与高度测量混在一起，可能导致占位高度异常变大。v3 已移除那部分高风险做法。

## 8. 配置参数说明

下面是常见配置项及推荐理解。

## 8.1 `enabled`
是否启用插件。

- `true`：启用
- `false`：禁用

禁用后，插件不再主动做虚拟化或恢复。

---

## 8.2 `showDebugOverlay`
是否显示右上角调试面板。

建议：
- 调试阶段：开启
- 日常使用：可关闭

---

## 8.3 `aggressiveMode`
是否使用更激进的窗口化策略。

含义：
- 开启后：更积极地减少 live 消息数量
- 关闭后：更保守，更接近原始页面体验

建议：
- 长会话明显卡顿：开启
- 页面结构不稳定或担心兼容性：关闭

---

## 8.4 `overscanMessages`
视口上下额外保留的缓冲消息数。

例如：
- `overscanMessages = 2`
- 表示视口上方多留 2 条，下方多留 2 条

值越大：
- 恢复更平滑
- 但 live DOM 更多，优化效果变弱

值越小：
- 压缩 DOM 更激进
- 但滚动恢复更频繁

建议范围：
- `2 ~ 4`

---

## 8.5 `minMessagesForVirtualization`
至少达到多少条消息才开始窗口化。

作用：
- 避免对很短的对话做没有必要的虚拟化

值越大：
- 触发更晚
- 小中等对话可能完全不 virtualize

值越小：
- 触发更早
- 20 条左右对话也可能开始窗口化

v3 的默认思路更适合中等长度对话。

建议范围：
- `12 ~ 20`

---

## 8.6 `keepRecentMessages`
始终保留为 live 的最近消息数量。

目的：
- 保护当前对话底部区域
- 避免新近消息过早被虚拟化
- 降低输入、复制、继续追问时的干扰

值越大：
- 交互更稳
- 但优化效果减弱

建议范围：
- `3 ~ 6`

---

## 8.7 `freezeAfterInputMs`
输入后冻结虚拟化的时间，单位毫秒。

作用：
- 用户打字后，先暂停虚拟化
- 避免输入链和虚拟化链路同时抢主线程

值越大：
- 输入更稳
- 但窗口化响应更慢

值越小：
- 更新更积极
- 可能影响输入流畅度

建议范围：
- `500 ~ 1000`

---

## 8.8 `contentVisibilityOnly`（如果提供）
仅启用较安全的显示层优化，不做真正的 DOM 替换。

适合：
- 想先保守验证兼容性
- 对 placeholder 不接受
- 需要最大限度保留原始 DOM 行为

这种模式下，可能不会出现 `virtual > 0`，因为它并没有真正把消息替换为 placeholder。

---

## 8.9 `mode`
通常对应 UI 上展示的运行模式，例如：

- `safe`
- `aggressive`

它是对多组参数组合的概括显示，不一定是单独的底层布尔值，但便于快速理解当前策略。

---

## 9. 调试面板字段说明

### `messages`
当前识别到的总消息数。

### `live`
当前仍以真实 DOM 存在的消息数。

### `virtual`
当前被 placeholder 替代的消息数。

### `updates`
状态更新次数。包括初始扫描、滚动更新、输入后恢复、mutation 更新等。

### `reason`
当前最近一次更新的原因，例如：

- `init`
- `mutation`
- `scroll`
- `post-input`
- `manual-restore`
- `settings-change`

### `detector`
当前消息识别策略。

### `mode`
当前模式，如 `aggressive`。

## 10. 使用建议

## 10.1 推荐的起步配置
对大多数长对话，推荐先从下面的思路开始：

- `aggressiveMode = true`
- `overscanMessages = 2`
- `minMessagesForVirtualization = 12`
- `keepRecentMessages = 3`
- `freezeAfterInputMs = 700`

---

## 10.2 如何判断插件是否真的生效
不要只看 `messages > 0`。真正有效的标志是：

- `virtual > 0`
- 滚动时 `live / virtual` 会变化
- 点击 placeholder 可以恢复原消息
- 长对话滚动和打字更顺

---

## 10.3 如果 `virtual` 一直是 0
常见原因：

1. 对话消息数还没达到最小阈值
2. 刚打完字，仍处于 `post-input`
3. `overscan` 太大
4. `keepRecentMessages` 太大
5. 当前停留位置可见消息本来就很多
6. 处于 `contentVisibilityOnly` 等保守模式

---

## 10.4 如果页面上出现很多灰色 placeholder
这通常表示：

- 窗口化已生效
- 旧消息已被摘出 DOM
- 可点击其中某条恢复查看

如果出现恢复不了的情况，说明插件逻辑异常，建议：

- 刷新页面
- 临时关闭激进模式
- 重新加载扩展
- 保留 debug overlay 继续观察

---

## 11. 内存与上下文说明

## 11.1 被虚拟化的消息还在不在内存里？
在 v3 的正常工作路径下：

- **通常仍在插件内存里**
- 只是暂时不在页面 DOM 中

这也是点击恢复能够工作的原因。

---

## 11.2 会不会影响 GPT 的正常上下文？
通常不会。

原因：
- ChatGPT 的模型上下文不依赖页面是否把所有历史消息都保留在 DOM 中
- 插件优化的是显示层，而不是服务端会话层

但要注意，依赖完整 DOM 的网页功能可能受影响，例如：

- 浏览器页内搜索旧消息
- 某些翻译扩展
- 某些全文复制/导出逻辑

---

## 12. 已知限制

- 页面 DOM 结构如果发生较大变化，消息识别策略可能需要更新
- 被 virtualize 的消息在当前时刻不参与浏览器全文搜索
- 某些依赖完整 DOM 的扩展可能表现异常
- 页面刷新后，插件内存状态会重建
- 如果 ChatGPT 官方前端更新较大，可能需要重新调 detector 逻辑

---

## 13. 排障指南

## 13.1 `messages: 0`
说明插件没识别到消息节点。

排查：
- 刷新页面
- 看是不是进入了非标准对话页
- 更新 detector 选择器
- 打开 debug overlay 观察 `detector`

---

## 13.2 `messages > 0` 但 `virtual = 0`
说明识别到了消息，但还没开始窗口化。

排查：
- 降低 `minMessagesForVirtualization`
- 降低 `overscanMessages`
- 降低 `keepRecentMessages`
- 等 `post-input` 结束再看
- 换到更长的对话测试

---

## 13.3 出现灰块但恢复不了
v3 理论上已修复主要恢复链路问题。

排查：
- 重新加载扩展
- 关闭其他强干预 DOM 的扩展
- 刷新页面后再试
- 观察是否有异常 detector 或重复 mutation

---

## 13.4 输入仍卡
可能原因：

- 当前会话仍然太大
- 其他扩展也在处理输入或 DOM
- `freezeAfterInputMs` 太短
- `keepRecentMessages` 太大导致底部仍很重

建议：
- 延长 `freezeAfterInputMs`
- 再减少 `overscanMessages`
- 临时禁用翻译/划词类扩展对比测试

---

## 14. 后续可扩展方向

可以继续演进的方向包括：

- 仅对极老消息做更激进的虚拟化
- placeholder 内保留一小段纯文本摘要
- 支持按消息类型区别策略（代码块 / 图片 / 普通文本）
- 更智能的恢复批量调度
- 对项目页、分享页做单独 detector
- 增加性能统计，例如长任务、滚动恢复耗时、平均 live 节点数

---

## 15. 总结

这个插件的核心思想很简单：

> 不去改 ChatGPT 的模型能力，而是把超长聊天页面当成一个“长列表”，通过窗口化 DOM 减少浏览器在输入、滚动和布局上的负担。

如果你把它理解为：

- **显示层窗口化**
- **会话层不删除**
- **历史消息按需恢复**

那就基本理解对了。

## 16. 版本说明（v3）
v3 的重点改进：

- 修复虚拟化后消息失联
- observer 忽略插件自身的内部 DOM 替换

- 去掉高风险的高度膨胀逻辑
- 默认参数更适合 20 条左右对话开始窗口化
